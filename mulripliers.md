Отличный вопрос! Давайте детально проанализируем архитектуру системы мультипликаторов. Это действительно критический момент, который нужно хорошо продумать.

## 🤔 **Анализ проблемы**

### **Три типа мультипликаторов**

#### 1️⃣ **Мультипликаторы по отчёту** (исторические на дату отчёта)
- **Когда:** На момент публикации финансового отчёта
- **Источник цены:** `FinancialReport.price_per_share` (цена на дату отчёта)
- **Финансовые показатели:** Из самого отчёта (revenue, net_income, equity...)
- **Неизменность:** Застывшие во времени, не меняются
- **Пример:** P/E = 15.3 на 31.12.2023 (по отчёту за Q4 2023)

#### 2️⃣ **Актуальные мультипликаторы** (свежие, "живые")
- **Когда:** Сейчас, в реальном времени
- **Источник цены:** **Текущая цена акции** (из API биржи)
- **Финансовые показатели:** Из **последнего** финансового отчёта
- **Изменчивость:** Меняются каждую секунду вместе с ценой
- **Пример:** P/E = 18.7 сейчас (цена выросла, но отчёт тот же)

#### 3️⃣ **Исторические мультипликаторы** (временной ряд)
- **Когда:** Каждый день/час за период
- **Источник цены:** Исторические цены акции
- **Финансовые показатели:** Из отчёта, актуального на ту дату
- **Назначение:** Графики, анализ трендов, сравнение
- **Пример:** P/E за последний год (365 точек для графика)

---

## 📊 **Проблемы и вопросы**

### **Проблема 1: Цена акции**
У нас сейчас `price_per_share` только в `FinancialReport` — это цена **на дату отчёта**.

**Вопросы:**
- Где хранить текущую цену акции?
- Как обновлять её в реальном времени?
- Нужна ли отдельная таблица `stock_prices` с историей?

### **Проблема 2: Частота обновления финансовых показателей**
Отчёты выходят раз в квартал/год, но цена меняется постоянно.

**Вопросы:**
- Какие показатели брать для актуальных мультипликаторов? (Последний отчёт? TTM - trailing twelve months?)
- Как обрабатывать период между отчётами?

### **Проблема 3: Хранение vs вычисление**
**Вариант А:** Хранить в БД
- ➕ Быстрый доступ
- ➕ История всегда доступна
- ➖ Много данных (1 компания × 365 дней × 6 мультипликаторов)
- ➖ Нужно регулярно обновлять

**Вариант Б:** Вычислять на лету
- ➕ Всегда актуально
- ➕ Не занимает место
- ➖ Медленнее
- ➖ Нужно постоянно дёргать API за ценами

**Вариант В:** Гибридный
- Мультипликаторы по отчётам — вычислять и **кэшировать** в БД
- Актуальные — вычислять на лету из API
- Исторические — хранить агрегированно (по дням)

## 🤔 **Ключевые вопросы для обсуждения**

### **1. Источник текущих цен акций**
- У вас уже есть интеграция с Tinkoff Invest API?
- Планируете обновлять цены автоматически (фоновая задача) или по запросу?
- Нужен ли real-time или достаточно цены закрытия дня?

### **2. Финансовые показатели для актуальных мультипликаторов**
**Варианты:**
- **Последний отчёт** (проще, но может быть устаревшим)
- **TTM** (Trailing Twelve Months — сумма за последние 4 квартала, точнее, но сложнее)
- **LTM** (Last Twelve Months — аналогично TTM)

### **3. Частота обновления исторических данных**
- Ежедневно в конце торгов?
- Только по рабочим дням?
- Ретроспективно заполнять при добавлении отчёта?

### **4. Стратегия кэширования**
**Варианты:**
- **Eager** — вычисляем и сохраняем сразу при создании отчёта
- **Lazy** — вычисляем при первом запросе, потом кэшируем
- **Scheduled** — фоновая задача раз в день/час

### **5. API эндпоинты**
Какие нужны:
```
GET /companies/{id}/multipliers/current     - Актуальные
GET /companies/{id}/multipliers/history     - История (для графиков)
GET /reports/{id}/multipliers               - По конкретному отчёту
GET /multipliers/compare?companies=1,2,3    - Сравнение компаний
```

## 💡 **Мои рекомендации**

**Для вашего проекта (Graham Analyzer) я бы предложил:**

### **Вариант 2 (средний) с поэтапной реализацией**

**Почему:**
1. Метод Грэма требует анализа истории — нужны графики
2. Но не нужна real-time точность — достаточно дневных данных
3. Вариант 1 слишком простой, Вариант 3 избыточный

**Структура таблиц:**
```
Company (есть)
  + current_price
  + price_updated_at

FinancialReport (есть)
  + computed: multipliers на дату отчёта

StockPrice (новая)
  - История цен по дням

Multiplier (новая, кэш)
  - Вычисленные мультипликаторы
  - Типы: report_based / daily / current
```

**Логика:**
- При создании отчёта → сразу вычисляем мультипликаторы и кэшируем
- Фоновая задача раз в день → обновляет цены и пересчитывает актуальные мультипликаторы
- API по запросу может пересчитать на лету если нужно
